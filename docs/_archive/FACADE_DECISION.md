# Facade Decision: Bitrix API & Package Structure

Настоящий документ фиксирует архитектурное решение по ролям модулей itrix24_api.py, itrix/ и itnewton_sync_to_api.py в рамках итерации «гигиена» проекта itrix24-automation.

## Текущее состояние (на момент итерации)

1. **bitrix24_api.py (Bitrix_API_Module)**
   - Является основной точкой входа к Bitrix24 REST API.
   - Реализует класс Bitrix24API с методами call, call_batch, get_all и 	est_connection.
   - Содержит базовую логику ретраев и обработки ошибок HTTP.
   - Используется большинством CRM-скриптов и основным пайплайном.

2. **Пакет bitrix/ (Bitrix_Package)**
   - itrix/api.py: Тонкий shim, выполняющий re-export класса Bitrix24API из itrix24_api.py для обратной совместимости.
   - itrix/recordings.py: Реализует специфичную логику поиска и разрешения URL записей звонков, используя Bitrix24API.
   - itrix/transcriptions.py: Содержит логику прикрепления расшифровок к звонкам.
   - Роль: вспомогательная логика, вынесенная в пакет для разгрузки корня.

3. **bitnewton_sync_to_api.py (Sync_Facade)**
   - Тонкая CLI-обёртка для интеграции с внешними .bat-файлами.
   - Инкапсулирует вызов основного пайплайна с передачей параметров.

## Принятые решения

1. **Канонический модуль:** itrix24_api.py официально закрепляется как основной низкоуровневый интерфейс к Bitrix24 API. Все новые высокоуровневые модули должны импортировать Bitrix24API напрямую из этого файла.
2. **Роль пакета bitrix/:** Пакет сохраняется для хранения специализированной логики (звонки, записи, расшифровки), которая требует декомпозиции. 
3. **Дублирование:** На текущий момент выявлено дублирование через re-export в itrix/api.py. Оно сохраняется для предотвращения поломок в коде, который может использовать rom bitrix.api import Bitrix24API.
4. **Удаление:** Удаление дубликатов и полный переход на единый интерфейс откладывается на следующие итерации рефакторинга. В рамках «гигиены» изменения ограничиваются логированием.

## План устранения дублирования (будущие итерации)

1. Инвентаризация всех вхождений импорта Bitrix24API.
2. Миграция всех потребителей на прямой импорт из itrix24_api.
3. Удаление itrix/api.py после подтверждения отсутствия зависимостей.
4. Оценка целесообразности переноса Bitrix24API внутрь пакета itrix/ с оставлением shim-файла в корне (инверсия текущей схемы).
